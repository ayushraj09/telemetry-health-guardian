Unit tests first (no MCP, no real LLM, no real OTLP -- everything
scheduler.py/main.py call out to is monkeypatched, same discipline as every
earlier stage's test file):
    python -m pytest guardian/tests/test_scheduler.py guardian/tests/test_main.py

These confirm: (1) AuditStore's service=None -> "_all_" keying and
get/set/all round-trip; (2) run_audit_cycle calls R1/R2/R3/R6 via the (fake)
MCP client, combines+scores them, generates a narrative, writes back when a
writeback is given, and caches the result -- and that a narrative-generation
failure degrades to narrative_error being set rather than failing the whole
cycle; (3) build_scheduler registers the "guardian-audit-loop" job at the
configured interval; (4) every FastAPI route's happy path and its
SigNoz/LLM-failure error-status mapping (502), including /chat's
run-a-fresh-audit-if-nothing-cached fallback and /audit/report's 404 when
nothing has run yet.

Then, against the live stack (needs .env fully configured -- SigNoz Cloud
MCP + at least one LLM provider from Stage 5, same as every stage since):

1. Start the backend: "uvicorn guardian.main:app --reload --port 8000" from
   the repo root. Watch its startup logs -- confirms (a) alert provisioning
   ran or was skipped per GUARDIAN_ALERT_WEBHOOK_URL, (b) the audit loop
   started, (c) the first background audit cycle logged a health score.
2. (Optional, for the live demo) start the frontend in a second terminal:
   "streamlit run frontend/app.py". Set the sidebar's Service field, click
   "Run audit now," confirm the Health Report tab renders a score + per-rule
   finding counts + narrative, then ask a question in the Chat tab.
3. Copy experiments/test_stage7.py to the repo root (same pattern as every
   earlier stage's driver) and run the Section 8 flow end-to-end:
     - clean baseline: run demo-agent-app/app.py with no chaos, then
       "python test_stage7.py --step baseline"
     - chaos run: rerun app.py with CHAOS_MODE=1 and all four required rates
       at 1.0 (same command Stage 6's guidance uses), then
       "python test_stage7.py --step chaos"
   Confirm the chaos run's printed health score is visibly lower than the
   baseline run's, GET /audit/report agrees with what POST /audit/run just
   cached, and the /chat answer to "why did the research pipeline's score
   drop?" names every rule that actually fired (the script's own
   rules_fired_but_uncited check is a mechanical proxy for this -- still
   read the printed answer, same caveat test_stage5.py's guidance already
   noted for validate_citations).

Gate criterion (Section 6, Stage 7): "does the full verification flow
(Section 8) run end-to-end without manual intervention?" -- concretely, all
seven of Section 8's numbered checks pass against a single baseline+chaos
pair of demo-agent-app runs, driven only by the commands above (no
hand-editing SigNoz, no manually reading raw MCP tool output to work around
a broken endpoint).

Only after this gate passes, and only if instructed to continue, does
Section 6 permit starting R7 (Stage 7's second half) -- do not begin
