3. In the SigNoz UI, go to Traces, filter by your service name, and open the trace from that run.
    Check the three literal gate criteria:
        (i) Token counts present — expand a chat <model> span, confirm gen_ai.usage.input_tokens / gen_ai.usage.output_tokens show real (non-zero) numbers.
        (ii) Model name present — same span, confirm gen_ai.request.model matches what you actually called (e.g. gpt-4o-mini).
        (iii) One connected trace tree — the whole run (Planner span → Fetch&Read → Fact-Check's parallel tool-call spans → Report Writer) should appear as a single trace with a single root, not split into multiple disconnected traces. Pay particular attention to the Fact-Check spans specifically — those are the ones that would go orphaned if the ThreadPoolExecutor.submit patch weren't working.
    Repeat for 2 more runs (different questions/PDFs) — the spec asks for 3 sample traces, since one clean run could be a fluke and won't catch e.g. an intermittent context-propagation miss under real concurrency/timing (which is exactly the kind of bug that behaves differently than in my synthetic in-memory test).
